Team 前端开发规范
2019-07-28 · 前端 · 字数 3024 · 阅读预计 22 分钟

前言

本厂规定的前端开发代码规范风格要求，算是最合理的 Javascript/Typescript/React/JSX 编码规范之一了。

具备了：

  • javascript 风格指南。
  • react 风格指南。
  • less/sass 风格指南。

版本：v1.0.0
更新至：2019-07-29
编写：Lizhooh

说明

此规范请大家务必遵守规则，如有犯错，请接受相关处理。

javascript 规范

javascript 规范比较多，请阅读这篇文章：https://github.com/yuche/javascript

部分变化以 ESLint 的规则配置为准，大部分情况下在 vscode 上格式化代码（ctrl + shift + f），可以达到一个良好的代码。

注意：代码缩进是 4 个空格。

变量命名

变量命名使用 骆驼式风格 命名，不用出现 _ 的写法。

对于类名的命名、类型的命名请使用大写开头。
对于接口的命名，请以 I 口头。

空格与换行

在大部分的情况下都要加空格，变量，操作符，控制流程，()，{} 等。

大部分情况下在 vscode 上格式化代码（ctrl + shift + f）可以解决空格的问题。

关于换行，函数，控制流程都要换行，if 在只有一个语句时可以不换行。

空行

在大多情况下时不用加空行的，有时候为了能够突出显示，添加一行空行，让代码有一定的区分度。

从 node_module 里导入的模块和自己编写的模块，需要添加一行空行加以区分（必须这样）。

当 import/require 很多模块时，要求添加空行加以区分，根据类别（建议）。

在函数里，定义变量之后添加空行加以区分（建议）。

一些变量非常多的情况下以一行空行以便于区分。

react 规范

由于使用 React + Typescript 开发，因此多多少少掺杂着一些 Typescript 的风格。

命名规范

  • 文件扩展名：react 组件文件使用 .tsx 扩展名。
  • 文件名: 文件名使用帕斯卡命名，如 ReservationCard.tsx。
  • 引用命名：react 规定组件需要大写，因此也是使用帕斯卡命名，如 <ReservationCard />。

  • 模块命名：如果整个文件夹是一个模块，使用 index.tsx 作为入口文件，其余的是归属的组件。

  • 高阶组件命名：HOC 组件使用 withXxx 的命名方式，以便于区分是否是高阶组件函数。

空格与换行

在组件的属性之间只保留一个空格，在闭合符前保留一个空格。

组件的属性 {} 两边不要留空格。

变量与属性

组件的属性规范：

  • JSX 属性名使用骆驼式风格 camelCase，不能出现 - 和 _。
  • 如果是 render props，请使用 renderXxxx 的命名方式。
  • 如果是回调函数，请使用 onXxxx 的命名方式。
  • 如果是布尔类型，建议是 isXxxx，hasXxxx 的命名方式，并且不能缺省。
  • 字符串属性使用单引号，而不是双引号。
  • 如果是常量字符串，则不要使用 {'abc'} 这种形式。

对齐与括号

JSX 分为单标签和双标签，其对齐方式请遵循：

  • 单标签能一行显示不需要换行（不过长情况，大致是 80 个字符或者在整体里显得不那么突出）。
  • 单标签过长时请换行，换行的风格要求尾部单独一行。
  • 双标签能一行显示不需要换行（不过长情况，大致是 80 个字符或者在整体里显得不那么突出）。
  • 双标签过长时请换行，换行的风格要求尾部不换行。

对于有默认项的组件，请提供完整的默认项。

类组件函数顺序

按照 react 组件的声明周期函数顺序，唯一例外的是 render 函数需要放在最后面。

  • 静态方法/属性放在最前面。
  • 构造函数放在最前面，但是在静态方法/属性后面。
  • 类的属性/方法放在构造函数后面。
  • 如果使用了 readonly state，将其放在构造函数后面。
  • 在构造函数之后开始是组件的生命周期函数。

state 与 props 的使用

在类组件里的，state 和 props 要求在函数顶层进行解构（在 render 函数里必须这样做）。

顺序是解构 this.props 在前面。

如果存在 this.props 和 this.state 的结构情况，那么要在 return 之前空一行。

如果遇到重名的问题，请使用别名。

对于函数组件，请在函数参数里进行解构（如果不想解，请看下面的方式）。

如果因为某些事情，无法在参数里解构，那么要在函数的顶层进行解构。

如果 props/state 解构过长（相比整体觉得过长），请换行。

同时 state/props 的解构要保持有序性，最好将回调函数放在后面。

在解构之后，可以提供默认值的，其默认级别比 defaultProps 要高，但是不建议在解构里提供默认值，而是通过 defaultProps。

hooks 函数

hooks 函数的规范，请根据官方的要求。

规范：

  • 只在函数组件最顶层使用 Hook。
  • 不要在循环，条件或嵌套函数中调用 Hook。
  • 不要在普通的 JavaScript 函数中调用 Hook。
  • 可以在自定义 hooks 里使用 hooks，但是命名要使用 useXxxx 的命名方式。
  • hooks 函数应该内容简单，只涉及逻辑处理，特别情况下还涉及组件的处理。

less/sass 规范

注意：代码缩进是 4 个空格。

命名规范

less/sass 的本质还是 css，因此，命名只能是小写，使用 - 进行分割，不允许出现 大写，_。

属性的命名也是全小写。

对于 less/sass 的变量命名也是如此。

空格

属性与值自己需要添加空格。

换行

样式的编写，当只有一个属性时可以不用换行和写结束分号，其余的情况都要换行。
