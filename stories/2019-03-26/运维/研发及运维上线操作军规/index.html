研发及运维上线操作军规
2019-03-26 · 后端 · 字数 2069 · 阅读预计 16 分钟

前言

为了保证生产环境的持续、稳定、高效地运转，并且使新同学更快的掌握线上操作的基本方法，编写本文加以约束。

名称解析

  • OP：运维人员
  • RD：研发人员
  • FE：前端开发人员
  • BE：后端开发人员
  • QA：测试人员
  • PM：产品经理
  • DBA：数据库管理员

1. 线上操作

条款 1.1：禁止流量高峰时期，进行影响 cache 的升级

  • 内容：对影响 cache 的升级操作禁止在流量高峰进行。
  • 正确：应该在服务流量低峰期进行上线或操作。
  • 说明：减少上线或操作对用户的影响，在异常时候减少损失。

条款 1.2：禁止程序线上“裸奔”

  • 内容：禁止程序在线上“裸奔”。
  • 正确：应该在程序上线前增加相应的“监控”，“统计”等。
  • 说明：防止服务异常时，OP 无法知晓，不能及时处理。

条款 1.3：禁止上线程序没有“回滚方案”

  • 内容：禁止程序上线时没有“回滚方案”。
  • 正确：所有上线必须有回滚方案。
  • 说明：防止因回滚准备不足，导致回滚时间变长或出错。

条款 1.4：禁止不进行备份

  • 内容：禁止不进行备份。
  • 正确：上线前，至少备份其变更部分。
  • 说明：避免回滚时无法快速进行有效回滚。

条款 1.5：禁止无依据操作

  • 内容：禁止无依据的操作。
  • 正确：依据已有的 checklist，依据 OP 和 RD 讨论的结论进行。
  • 说明：避免由于个人单点，对服务处理不当，导致上线操作出现非预期的结果。

条款 1.6：机器间拷贝数据应限速

  • 内容：进行线上机器间数据拷贝之前，要查看系统性能、负载，分析和判断后进行。
  • 正确：数据传输必须进行限速传输。
  • 说明：避免大数据操作对系统影响。

条款 1.7：禁止程序上线后不复查

  • 内容：禁止程序上线后不复查。
  • 正确：程序上线后，应检查服务资源消耗、服务错误日志。

条款 1.8：禁止 RD 不按流程上线

  • 正确：需要经过审核后，通过 deploy 上线。
  • 说明：防止 RD 误更新导致服务异常。

条款 1.9：禁止 RD 修改服务资源配置

  • 内容：修改涉及机器 nginx 等配置，OP 要 review 后才能上线。
  • 正确：评估 RD 对配置的修改，然后做出上线判断。
  • 说明：一个服务的资源配置修改会影响到其他服务，甚至整个集群，对配置修改上线需要严格控制。

条款 1.10：建议在特定时间上线

  • 内容：下午 6 点后，尽量不上线，尽量不在周六上线。
  • 正确：尽量在周一至周五的 11-12 点、17–18 点进行上线操作。
  • 说明：在工作时间进行上线，出现问题，能够及时找到相关人员进行解决。

2. 定时任务

条款 2.1：不建议整点运行任务

  • 内容：不建议在整点时间运行任务，如 00:00:00。
  • 正确：应该跳过整点运行，如 00:00:01。
  • 说明：避免因系统时钟提前运行或不运行导致任务执行异常。

条款 2.2：禁止程序屏幕打印日志

  • 内容：不允许程序屏幕打印日志。
  • 正确：应该将输出重定向到其他文件，或以 &>/dev/null 结尾。
  • 说明：避免系统将标准输出，打到 /var/spool/clientmqueue/ 下，引起根分区空间不足，导致硬盘报警。

条款 2.3：禁止定时任务“裸奔”

  • 内容：不允许定时任务“裸奔”。
  • 正确：应该对定时任务程序进行监控。
  • 说明：避免因程序异常退出，不能及时处理，导致任务不按时执行。

条款 2.4：禁止 crontab 中直接写 shell 命令

  • 内容：不准在 crontab 中直接写 shell 命令（cd 命令除外）。
  • 正确：如果有需求，必须用脚本封装来实现。
  • 说明：避免由于 crontab 解析问题引起运行异常。

3. 日志方面

条款 3.1：日志文件须切割

  • 内容：日志文件须切割。
  • 正确：日志文件必须切割处理。常用的切割方法包括 cronlog/定期 mv 等。
  • 说明：大日志文件读写性能低下，防止服务程序日志过大引起服务故障，磁盘空间不足，引发程序错误。

条款 3.2：禁止线上进行资源高消耗的操作

  • 内容：线上分析日志时，如果预期要消耗大量资源，不直接在线上操作。
  • 正确：将日志限速拖到线下再分析。联系 OP 将机器流量摘掉再进行操作。
  • 说明：系统资源消耗过多，影响线上服务，导致翻车。

条款 3.3：禁止对 IO 敏感服务使用重定向方式文件覆盖

  • 内容：对于 IO 敏感的服务，禁止使用重定向的方式进行文件覆盖。
  • 正确：mv 后，低优先级删除的方式进行。（nice -n **）或者停服务再删。
  • 说明：防止 IO 抢占影响服务。

4. 文件修改

条款 4.1：禁止直接修改正在运行的程序的配置，数据。

  • 内容：对于正在运行的程序，其配置，数据，程序文件禁止直接修改。
  • 正确：应该修改其副本，diff 确认正确后，再进行替换。
  • 说明：防止程序重启或重载时加载正在修改的文件。

条款 4.2：禁止线上直接修改程序代码

  • 内容：禁止在线上直接修改程序代码。
  • 正确：应该走程序的上线流程来进行上线操作。

5. 安全方面

条款 5.1：禁止裸奔 SQL 语句

  • 内容：禁止裸奔 SQL 语句，必须做 XSS 及 SQL 注入过滤。
  • 正确：使用 ROM 框架进行数据库操作，如果直接写 SQL 语句则需要对字符串进行过滤。
  • 说明：预防 XSS 注入，及 SQL 注入攻击。

条款 5.2：添加反爬虫机制

  • 内容：添加反爬虫机制，预防被爬虫大量抓取数据，甚至是拖库。
  • 正确：添加部分反爬虫机制，限制 API 输出的数据格式。

条款 5.3：禁止使用“明码”存储的方式设计账号模块

  • 内容：禁止使用“明码”存储的方式设计账号模块。
  • 正确：用户账号密码存储应当使用特定的算法（sha256）和加 salt 进行加密存储。
  • 说明：预防数据库被破解后账号密码涉密。

条款 5.4：管理员禁止使用过于简单的密码

  • 内容：管理员禁止使用过于简单的密码，例如 root123，abc12345，123456asdf，admin123 等过于简单的密码。
  • 正确：使用随机字符串生成工具生成 8 位或 16 位密码，并做备份记录。

条款 5.5：永远也不要相信客户端的输入

  • 内容：永远也不要相信客户端的输入，必须做格式校验。
  • 正确：对客户端的输入进行格式、类型的校验。
  • 说明：预防数据格式、类型不正确，导致程序运行出错。

6. Root 用户

条款 6.1：慎用 kill -g 命令

  • 内容：慎用 killall -g 命令。
  • 正确：同源程序会被错误杀掉，如 a 和 b 都由同一个 supervise 调用，就会出问题。

条款 6.2：禁止使用 rm –rf * 进行文件删除

  • 内容：禁止使用 rm –rf * 进行文件删除
  • 正确：建立一个 trashcan 目录，删除文件时使用 mv 移动到 trashcan 里，每个星期 OP 对 trashcan 里的文件进行 rm 操作。
  • 说明：避免失误操作 rm -rf / 造成的惨剧（血案）！

7. 故障处理

条款 7.1：当发生故障时，首先保证服务可用

  • 内容：当服务出现故障，一定先保证线上服务可用，再进行定位。
  • 正确：可以采取的方案包括将部分机器摘掉以保存现场，手动备份日志等。
